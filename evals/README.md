# Evals

A three-arm harness that measures the token compression of this repo's
skills by running the same prompts through Claude Code and comparing the
generated output token counts.

**No snapshot is committed to this repository.** The harness reports only
on data you generate yourself with `llm_run.py` (see "Provenance" below).
Until you run it, this directory makes no claims about TLDR's compression
numbers — for the honest cost/benefit picture, see
[`docs/HONEST-NUMBERS.md`](../docs/HONEST-NUMBERS.md).

## The three arms

| Arm | System prompt |
|-----|--------------|
| `__baseline__` | none |
| `__terse__` | `Answer concisely.` |
| `<skill>` | `Answer concisely.\n\n{SKILL.md}` |

The honest delta for any skill is **`<skill>` vs `__terse__`** — i.e.
how much the skill itself adds on top of a plain "be terse" instruction.
Comparing a skill to the no-system-prompt baseline conflates the skill
with the generic terseness ask, which inflates the numbers.

## Why this design

- **Real LLM output**, not hand-written examples (no circularity).
- **Same Claude Code** the skills target — no separate API key.
- **Control arm** isolates the skill's contribution from the generic
  "be terse" effect.
- **Metadata embedded in every snapshot** (model, CLI version, timestamp)
  so any number you quote is auditable.

## Provenance

The harness code (`llm_run.py`, `measure.py`, `plot.py`) is derived from
the [caveman](https://github.com/JuliusBrussee/caveman) eval harness (MIT);
see [`docs/legal/ATTRIBUTION.md`](../docs/legal/ATTRIBUTION.md). An earlier
revision of this repository committed caveman's own 2026-04-08 snapshot
with arm names relabeled as if it were a TLDR run. It was not, and it has
been removed — see [`snapshots/README.md`](snapshots/README.md). Never
commit a snapshot that was not generated by this harness against this
repository's skills.

## Files

- `prompts/en.txt` — fixed list of dev questions, one per line.
- `llm_run.py` — runs `claude -p --system-prompt …` per (prompt, arm),
  captures real LLM output, writes `snapshots/results.json` along with
  metadata (model, CLI version, generation timestamp).
- `measure.py` — reads the snapshot, counts tokens with tiktoken
  `o200k_base`, prints a markdown table with median / mean / min / max /
  stdev across prompts. Prints a notice and exits if no snapshot exists.
- `snapshots/` — local output directory. Nothing is committed here.

## Generate a snapshot (requires `claude` CLI logged in)

```bash
uv run python evals/llm_run.py
```

This calls Claude once per prompt × (N skills + 2 control arms). Use
a small model to keep it cheap:

```bash
TLDR_EVAL_MODEL=claude-haiku-4-5 uv run python evals/llm_run.py
```

## Read the snapshot (no LLM, no API key)

```bash
uv run --with tiktoken python evals/measure.py
```

## Adding a prompt

Append a line to `prompts/en.txt`, then regenerate the snapshot.

## Adding a skill

Drop a `skills/<name>/SKILL.md`, then regenerate the snapshot. `llm_run.py`
picks up every skill directory automatically.

## What this does NOT measure

- **Fidelity** — does the compressed answer preserve the technical
  claims? A skill that replies `k` to everything would score −99% and
  "win". A future v2 could add a judge-model rubric.
- **Latency or cost** — out of scope. Note that skills add input tokens
  on every call, so output savings are not the full economic picture.
- **Cross-model behavior** — only the model used to generate the
  snapshot is measured.
- **Exact Claude tokens** — `tiktoken o200k_base` is OpenAI's BPE and is
  only an approximation of Claude's tokenizer. Ratios between arms are
  meaningful; absolute numbers are approximate.
- **Statistical significance** — single run per (prompt, arm) at default
  temperature. The min/max/stdev columns let you eyeball whether a
  number is solid or noisy, but this is not a powered experiment.
